Rules

What you must never do and what to use instead

Never use these

git pull
Runs fetch + merge silently. Creates a hidden merge commit on your branch. The #1 cause of messy histories.
Use this instead:
git fetch then git rebase origin/develop
VS Code "Sync Changes" button
Does pull + push in one click silently. You cannot see what it does before it does it.
Use this instead:
Open terminal (Ctrl+`) and run every step manually
VS Code Pull button (down arrow)
Identical to git pull. Creates a hidden merge commit.
Use this instead:
git fetch then git rebase origin/develop
git reset --soft on a shared branch + git add .
Un-commits everything including teammates' work into your working directory. One wrong git add . and you commit someone else's code as your own. Only safe if you are the ONLY person who has ever committed to this branch and even then, use rebase -i when in doubt.
Use this instead:
git rebase -i HEAD~N -- only touches your own commits
git push --force
Overwrites remote with no safety check. Can silently delete teammates' commits.
Use this instead:
git push --force-with-lease
git reset --hard
Permanently deletes all uncommitted file changes. No undo possible.
Use this instead:
git stash -- saves everything, restore anytime with git stash pop
VS Code "Discard All Changes"
Same as git reset --hard. Permanently wipes all uncommitted work instantly.
Use this instead:
git stash -- or right-click individual files and discard only those

IDE -- what's safe and what's not

Safe to use

Never click

Golden pattern: Terminal (prepare) then IDE (stage your files + commit) then Terminal (push, verify).

Local Repository
Saved commits on your machine. Nobody sees these until you push. Safe to experiment here freely.
Local-only operations
rebase -i HEAD~N -- squashes only YOUR commits. Surgical. Safe in teams. Count N first with: git fetch origin && git rev-list --count origin/develop..HEAD
rebase onto develop -- replays your ONE clean commit on top of latest develop. Do AFTER squashing. One conflict round max.
checkout -- switch branches locally. Nothing goes to origin.
stash -- saves uncommitted work safely. Restore with stash pop.

You only ever push to YOUR feature branch. Develop is read-only reference. To accidentally push to develop you would have to explicitly type git push origin mvp/develop which we never do.

Why squash before rebase -- the key insight

Rebase replays commits one by one on top of develop. If you have 6 WIP commits, it does 6 replays and each one can produce a conflict on the same file, over and over. That is not a bug, that is how rebase works mechanically.

WITHOUT squashing first
6 commits, 6 replays
6 possible conflict rounds
Same file, same conflict, 6 times
Exhausting and error-prone
WITH squashing first
1 commit, 1 replay
1 possible conflict round max
Clean, predictable, fast
The right order every time
The correct order is always: squash first, then rebase. Never rebase first with many commits.
