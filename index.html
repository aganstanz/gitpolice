Git Guide

The exact process for clean PRs — and what never to do

The correct order — every single time

1. Stash
→
2. Fetch
→
3. Squash
→
4. Rebase
→
5. Push
→
6. Stash pop
Why Squash before Rebase? — If you have 5 commits and rebase without squashing first, Git replays each commit one by one. Each one can cause a separate conflict. Squash into 1 commit first, then rebase — maximum 1 conflict, done in minutes.
Why Rebase at all? — Squashing without rebasing means your commit sits on old develop. Rebase moves it to the very tip of the latest develop so your PR applies cleanly.
Your base branch is never touched

Every command in this guide operates only on your feature branch. origin/develop is always used as a read-only reference — never written to.

git reset --soft origin/develop — moves only YOUR branch pointer back. develop is untouched.
git rebase origin/develop — replays YOUR commits on top of develop. Only your branch moves.
git push origin feature/your-branch --force-with-lease — pushes only the branch you name. develop is completely unaffected.

Never use these — and what to do instead

git pull
Secretly runs git fetch + git merge behind the scenes. Creates a hidden merge commit you never asked for. Completely skips Squash and Rebase. Your PR history becomes unreadable with all your WIP commits exposed.
Use this instead:
stash → fetch → squash → rebase → push → stash pop
VS Code "Sync Changes" button
Does pull + push in a single click silently. It skips Squash entirely, skips Rebase entirely, and pushes all your messy work-in-progress commits raw to origin.
Use this instead:
Open the terminal (Ctrl+`) and run every step manually
VS Code Pull button (the down arrow)
Identical to git pull. Same hidden merge commit. Same skipped Squash and Rebase.
Use this instead:
stash → fetch → squash → rebase → push
git add . after soft reset
After a soft reset, your branch may contain other people's commits that got mixed in. Running git add . blindly stages ALL of those files — not just yours. You will commit other people's work as your own.
Use this instead:
git reset HEAD . → stage ONLY your files in VS Code or IntelliJ → git checkout -- .
git push --force
Overwrites the remote branch without checking anything first. Can silently delete teammates' commits with no warning and no recovery.
Use this instead:
git push --force-with-lease
This checks the remote first. If something has changed that you didn't expect, it fails safely instead of deleting it.
git reset --hard
Permanently deletes all uncommitted file changes. There is no undo. Hours of work gone instantly.
Use this instead:
git stash
Stash saves everything safely. You can always restore it with git stash pop.
VS Code "Discard All Changes"
Same as git reset --hard. Permanently wipes all uncommitted work with no recovery.
Use this instead:
git stash — or discard individual files only, never all
Rebase without squashing first
If you have 5 commits, Git replays each one separately. You can hit a conflict at each commit — 5 commits means up to 5 separate conflict stops.
Use this instead:
Always squash into 1 commit first, then rebase — maximum 1 conflict
Rebase with leftover unstaged files still present
After squashing selectively, leftover unstaged files from other people's work will still be sitting in your working directory. Running git rebase immediately fails with: "error: cannot rebase: You have unstaged changes."
Use this instead:
git stash push -m "unwanted-files-temp" before rebasing, then drop it after

The golden pattern for syncing IDE and terminal: Terminal first (prepare) → IDE (stage your files + commit) → Terminal (continue, push, verify). Never do staging in IDE and terminal at the same time.

What each command actually does

git stash Saves ALL your current uncommitted file changes into a temporary pocket. Nothing is pushed anywhere. You can restore everything later with git stash pop.
git fetch origin Downloads information about what has changed on origin. Does NOT change any of your files. Does NOT change any branch. Purely informational download.
git reset --soft Removes one or more commits from your branch history but keeps all the file changes in your working directory. Used as the first step of squashing.
git add yourfile Moves a specific file into the staging area, marking it as "ready to be committed". Only use this with specific file paths, never git add . after a soft reset.
git commit Takes everything in the staging area and saves it as a permanent snapshot (commit) in your local repository. Nothing is sent to origin yet.
git rebase origin/develop Takes your commit(s) off your branch, moves the branch pointer to match the latest develop, then re-applies your commit(s) on top. Only YOUR branch moves. Develop is untouched.
git push --force-with-lease Sends your local branch commits to origin. The --force-with-lease flag first checks that nobody else has pushed to your branch since you last fetched. If something changed unexpectedly, it fails safely.
git checkout branchname Switches your working directory to a different branch. Purely local — nothing is sent to origin. Always stash uncommitted work before switching.
git stash pop Restores the most recently stashed files back into your working directory, and removes that stash entry from the list.
git log --oneline Shows your commit history in a compact list. Use this after every important step to verify your commit is there and looks correct. This is always the source of truth.
git push -u origin branchname Pushes a brand new branch to origin for the first time and sets up tracking. After this, future git push commands on this branch automatically know where to push.
