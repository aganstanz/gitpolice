🛡️ SAFETY GUARANTEE — your base branch is NEVER touched
git stash — saves your uncommitted files locally. Does not touch any branch.
git fetch origin — downloads info only. Does not change any file. Does not touch develop.
git reset --soft origin/mvp/develop — only moves YOUR feature branch pointer back. develop is read-only here, used only as a reference point. It is NEVER modified.
git rebase origin/mvp/develop — replays YOUR commits on top of develop. Only your feature branch moves. develop is untouched.
git push origin feature/your-branch --force-with-lease — pushes ONLY your feature branch. develop on origin is completely unaffected.
✓ Every single command in this guide operates exclusively on your feature branch.
✓ origin/mvp/develop is used only as a read-only reference point — never written to, never modified.
✓ To accidentally push to develop you would have to explicitly run git push origin mvp/develop — which we never do.
✅ THE ONLY CORRECT ORDER — memorize this
1. STASH
→
2. FETCH
→
3. SQUASH
→
4. REBASE
→
5. PUSH --force-with-lease
→
6. STASH POP
Every single time. Before every PR. Before every push after rebase. This order never changes.
⚠️ Never skip SQUASH. If you rebase with multiple commits, you get multiple conflict stops. Squash first = max 1.
⚠️ Never skip REBASE. Squash without rebase means your commit sits on old develop. Rebase puts it on the latest tip.
🚫 STRICTLY BANNED — never use these commands or buttons
git pull MOST DANGEROUS
Secretly runs git fetch + git merge. Creates hidden merge commit. Pollutes history forever. Makes PR unreadable. Completely skips SQUASH AND REBASE.
✅ INSTEAD — do all steps:
git stash
git fetch origin
git reset --soft origin/mvp/develop   # squash — check exact branch name first!
git reset HEAD .   # unstage everything
git add your/file1 your/file2   # stage ONLY your files
git checkout -- .   # discard other people's files
git commit -m "feat: your feature"
git stash push -m "unwanted-files-temp"   # stash leftover files before rebase
git rebase origin/mvp/develop
git push origin feature/your-branch --force-with-lease
git stash drop stash@{N}   # drop unwanted stash (already in develop)
git stash pop   # restore YOUR original stash
VS Code "Sync Changes" button EXTREMELY DANGEROUS
Does git pull + git push in ONE CLICK silently. Skips SQUASH entirely. Skips REBASE entirely. Pushes all your WIP/fix/test commits raw.
✅ Instead → Use terminal (Ctrl+`). Run stash → fetch → SQUASH → REBASE → push → stash pop manually.
VS Code "Pull" button (↓ arrow)
Identical to git pull. Same hidden merge. Same skipped SQUASH and REBASE. Never click it.
✅ Instead → terminal: stash → fetch → SQUASH → REBASE → push → stash pop
git push --force
Overwrites remote without checking. Silently deletes teammates' commits. No warning. No recovery.
✅ Instead → git push --force-with-lease (checks first, fails safely)
git reset --hard
Permanently deletes ALL uncommitted file changes. Cannot undo. Hours of work gone.
✅ Instead → git stash (saves work) or git reset --soft HEAD~1 (removes commit, keeps files)
VS Code "Discard All Changes"
Wipes ALL uncommitted work permanently. No recovery.
✅ Instead → git stash to save, or discard individual files only
git add . after soft reset REAL WORLD DANGER
After git reset --soft origin/develop, your branch may have had other people's commits mixed in. Running git add . blindly stages ALL of those files — not just yours. You will commit other people's work as your own.
✅ INSTEAD — stage selectively:
git reset HEAD .   # unstage everything first
git add path/to/your/file1   # only YOUR files
git add path/to/your/file2
git checkout -- .   # discard everything else
git commit -m "feat: your feature"
Rebase without squashing first COMMON MISTAKE
If you have 5 commits and run git rebase origin/develop WITHOUT squashing first, Git replays each commit one by one. Each one can cause a separate conflict. SQUASH first → 1 commit → max 1 conflict.
✅ Always SQUASH first: git reset --soft origin/develop → stage only your files → git commit → THEN REBASE
Rebasing without stashing leftover files first REAL WORLD BLOCKER
After squashing selectively, you will have leftover unstaged files (other people's work that was on your branch). Running git rebase will immediately fail with "error: cannot rebase: You have unstaged changes."
✅ INSTEAD — stash leftovers before rebase:
git stash push -m "unwanted-files-temp"   # stash leftovers safely
git rebase origin/mvp/develop   # now rebase works
git push origin feature/your-branch --force-with-lease
git stash drop stash@{N}   # drop it — those files are already in develop
⚠️ REAL WORLD WARNINGS — things the guide originally missed
Branch name may differ from the guide
Always check the exact base branch name before running commands. It might be mvp/develop not mvc/develop. Run git branch -a to see all branches.
After soft reset you may see many staged files that aren't yours
This happens when other people's commits were on your branch before the reset. Those files are NOT yours. Never git add .. Always stage only your specific files manually.
VS Code Commit button may or may not have worked
Always verify with git log --oneline -5 after committing. If your commit message appears at the top, it worked. If not, commit again via terminal.
Never drop a stash without checking what's in it first
Run git stash list to see all stashes. Match by name before dropping. Use git stash drop stash@{N} where N is the correct number for the unwanted stash.
credential-osxkeychain warning on Windows is harmless
If you see git: 'credential-osxkeychain' is not a git command on Windows — ignore it. It's a Mac keychain tool not available on Windows. Your push still worked.
This guide originally missed the PUBLISH step
After creating a new branch with git checkout -b, you should publish it to remote with git push -u origin your-branch. This sets upstream tracking and backs up your branch. Now fixed in the Step Guide.
Hard to type file paths in terminal?
If you find it difficult to type file names/paths in the terminal for git add, you can: stash your work with git stash, then use VS Code's Source Control panel to stage files by clicking the + button next to each file, commit using the ✓ checkmark button, then continue with the remaining terminal commands (rebase, push, etc.).
✅ What to do — the correct workflow always
1
Start of day / before any work — git fetch origin to see what changed. Never pull. Never sync.
2
Before raising PR — stash → fetch → SQUASH (stage only YOUR files) → stash leftovers → REBASE → push → drop unwanted stash → stash pop. Always SQUASH BEFORE REBASE.
3
Creating a new branch — fetch first → checkout base → rebase base → checkout -b new-name → publish with git push -u origin new-name
4
Switching branches — stash or commit first, then fetch, then checkout. Never click sync.
5
Resolving conflicts during rebase — fix file → git add . → git rebase --continue
6
Stuck after clicking Pull/Sync? — Abort → stash → reset → start fresh. See 🆘 Rescue in Step Guide.

⬇ git fetch vs 🚫 git pull — the critical difference

✅ git fetch (safe — use this)
Downloads info about new commits from origin.
Your files stay exactly the same.
Nothing changes. Nothing moves. No merge. No rebase.
Just downloads knowledge of what changed.
$ git fetch origin
# nothing changes on your branch
# nothing changes on develop
# see all branches: git branch -a
🚫 git pull (BANNED)
Secretly runs git fetch + git merge.
Creates a hidden merge commit.
Skips SQUASH entirely.
Skips REBASE entirely.
PR becomes unreadable. All WIP commits stay.
# git pull secretly runs:
1. git fetch origin
2. git merge origin/develop ← HIDDEN!
# creates merge commit you can't see
# your 10 WIP commits stay forever

⚔️ Conflicts — what they look like and how to fix

<<<<<<< HEAD
your code
=======
their code
>>>>>>> origin/develop
Pick one or combine. Delete ALL markers (<<<, ===, >>>). Then:
$ git add .
$ git rebase --continue
# if too messy: git rebase --abort (starts over)
