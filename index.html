Git Guide

The correct process for clean PRs — and what never to do

The complete flow — cracked

⚡ The Elixir — Two phases, zero edge cases
DAILY DEV Commit freely as you work. When you have multiple WIP commits piling up, run git rebase -i HEAD~N to squash them into 1 clean commit locally. No develop involved. No mixed files. Completely safe anytime.
PR TIME You already have 1 clean commit from daily squashing. Just stash → fetch → rebase onto develop → push. The rebase is trivial — 1 commit replaying onto develop, one conflict at most.

Daily dev order — squash your own WIP anytime

Commit WIP freely
→
git rebase -i HEAD~N
→
1 clean local commit
N = how many of your commits to squash. Run git log --oneline and count the commits at the top that are yours. That number is N. This never touches develop, never touches other people's files, works anytime.

PR time order — one final move onto develop

1. Stash
→
2. Fetch
→
3. Rebase onto develop
→
4. Push
→
5. Stash pop
Because you already have 1 clean commit from rebase -i, there is no squash step needed here. Just rebase onto develop and push.
⚠ Why rebase -i is the safe squash — not soft reset

Soft reset squash (the old way) un-commits everything including other people's work into your working directory at once. If any file was touched by both you and a teammate, their changes appear mixed in with yours. You have to hand-separate lines — error-prone and risky.

git rebase -i HEAD~N only operates on YOUR commits. It never touches develop, never touches other people's files, and nothing lands in your working directory. It is completely safe to run anytime during development.

rebase -i to squash → rebase onto develop to sync. That's the complete answer.

Never use these — and what to do instead

git pull
Runs fetch + merge silently. Creates a hidden merge commit on your branch. Skips rebase and squash entirely. This is the #1 cause of messy histories on shared branches.
Use this instead:
stash → fetch → rebase onto develop → push → stash pop
▶ When is git pull actually okay? (expand) Exceptions
Default rule: never use plain git pull on any branch in this team workflow. The exceptions below apply only in very specific, low-risk solo scenarios. When in doubt, use fetch + rebase instead — it is always safe.
Situation Which branch Safe? What to check first
You are the only person on your feature branch, no teammates have checked it out, and you have no local commits that aren't on origin yet feature/your-branch OK with caution Run git status — must show "nothing to commit". Run git log --oneline origin/your-branch..HEAD — must show nothing. If either shows anything, do NOT pull — use fetch + rebase instead.
You want to update your local copy of develop just to read/reference it, and you are not currently on develop mvp/develop (local reference only) OK Make sure you are NOT on develop (git branch — asterisk must not be on develop). Switch to your feature branch first. Then: git fetch origin is actually better even here — pull is technically fine but fetch is cleaner.
You have local commits that haven't been pushed yet Any branch Never Pull will merge remote into your local commits, creating a merge commit. Use fetch + rebase.
A teammate has also pushed to your branch (e.g. during pair work) Any shared branch Never Pull creates a merge commit mixing both histories. Use fetch + rebase.
You are on mvp/develop and want to update it mvp/develop Never Develop should only ever move forward via PRs merged in the browser. Never pull, push, or commit directly to develop from your machine.
The safe version of pull: If you genuinely need to pull, use git pull --rebase instead of plain git pull. This runs fetch + rebase instead of fetch + merge, so no hidden merge commit is created. But on this team, fetch + rebase manually is still preferred because it gives you full control.
Full safe sequence when pull is okay (solo branch, no local commits):
# Step 1: Verify you are on YOUR branch
$ git branch
# Asterisk must be on your feature branch, NOT develop
# Step 2: Verify no local-only commits exist
$ git log --oneline origin/your-branch..HEAD
# Must return empty. If anything shows, stop — use fetch + rebase instead.
# Step 3: Verify working directory is clean
$ git status
# Must say "nothing to commit". If not, stash first.
# Step 4: Now pull is safe
$ git pull --rebase origin your-branch
OK — no merge commit created

Overwrites remote with no safety check. Can silently delete teammates' commits pushed after your last fetch.

🚨 Rescue / Reflog

Git never truly deletes commits — reflog is your ultimate safety net

🟣 What is git reflog?

git reflog is a local log of every position HEAD has been in — every checkout, commit, reset, rebase, merge, and cherry-pick. Even if you deleted a branch, hard-reset past commits, or had a rebase go wrong, those commits still exist in reflog for ~90 days. This is your ultimate undo tool. It only exists locally — it is not on origin.

Scenario 1 — You reset or rebased and lost commits

# Step 1: See every position HEAD has been at
$ git reflog
# Output looks like:
a1b2c3d HEAD@{0}: rebase finished: returning to refs/heads/feature/login
f4e5d6c HEAD@{1}: commit: feat: add login form
9g8h7i6 HEAD@{2}: commit: wip: started login
3j2k1l0 HEAD@{3}: checkout: moving from develop to feature/login
# Step 2: Find the commit you want to recover — copy its hash (e.g. f4e5d6c)
# Step 3a: Restore that commit as your new HEAD (keeps it on your branch)
$ git reset --soft f4e5d6c
# Your files and commits are back. Nothing is deleted.
# Step 3b: OR — recover it to a new branch so your current branch is untouched
$ git checkout -b recovery/my-lost-work f4e5d6c
OK Recovered commits are safe on a new branch
Your commits exist in reflog even after a hard reset, a bad rebase, or deleting a branch — as long as you haven't run git gc and it's been less than ~90 days.

Scenario 2 — You deleted a branch accidentally

# Find the last commit that was on the deleted branch
$ git reflog
# Look for the last commit message you remember. Copy its hash.
# Recreate the branch at that commit
$ git checkout -b feature/recovered-branch a1b2c3d
OK Branch is back with all its commits

Scenario 3 — Rebase went completely wrong

# First try to abort if still in progress
$ git rebase --abort
# If abort doesn't help — find the state just before the rebase started
$ git reflog
# Look for: "checkout: moving from ... to ..." or "rebase: start"
# Copy the hash of the commit just BEFORE the rebase started
$ git reset --hard THAT_HASH
WARNING: --hard resets working directory. Only use if you have no uncommitted changes.
# Safer version: use --soft to keep your files
$ git reset --soft THAT_HASH
OK Branch is restored to pre-rebase state
--hard vs --soft in recovery: Use --soft when you want your file changes preserved in the working directory. Use --hard only when you want to completely discard everything back to that commit and your working directory is already clean (or you don't care about losing uncommitted changes).

Scenario 4 — Stash pop went wrong or stash was dropped

# List all stashes including dropped ones (they stay in reflog)
$ git fsck --no-reflog | grep commit
# Or search reflog directly
$ git reflog stash
# Apply a stash by its hash without removing it from the stash list
$ git stash apply stash@{2}
OK Stash contents restored

Quick reference — reflog commands

# See full reflog
$ git reflog
# See reflog for a specific branch
$ git reflog show feature/login-page
# See what a specific HEAD position looked like
$ git show HEAD@{3}
# Recover to a specific reflog position
$ git reset --soft HEAD@{3}
# How long does reflog keep entries?
# Default: 90 days for reachable commits, 30 days for unreachable ones
# git gc cleans them — avoid running gc manually
Reflog is local only. If you clone fresh or lose your machine, reflog is gone. If you know you need a commit long-term, push it to a recovery branch on origin immediately after recovering it.
